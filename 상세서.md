# 전통문화 기념품 RAG 콘텐츠 생성 어드민 상세서

작성 기준일: 2026-07-01  
문서 목적: 본 문서는 전통문화 기념품 200종의 제품 스토리 지식베이스 구축, RAG 기반 텍스트 생성, AI 이미지 생성 워크플로우를 구현하기 위한 UI/UX 및 시스템 설계 기준 문서다. 개발자와 디자이너는 본 문서를 기본 구현 기준으로 사용한다.

### 2.3 외부 API

| API | 필수 여부 | 사용 목적 | 비고 |
|---|---:|---|---|
| Gemini API 또는 Vertex AI Gemini | 필수 | 제품 스토리 요약, 다국어 콘텐츠 생성, 구조화 출력 | 정부지원사업 보안 요건이 강하면 Vertex AI 계약 검토 |
| Gemini Embeddings | 필수 | 제품 스토리 청킹 후 임베딩 생성 | 벡터 저장은 Supabase pgvector |
| Gemini 2.5 Flash Image 또는 Google 권장 이미지 생성 모델 | 필수 | 제품 이미지 배경 합성, 전통문화 콘셉트 이미지 생성 | 운영 전 Google 공식 모델명 재확인 |
| Grounding with Google Search | 선택 | 생성 결과의 출처 기반 검증, 보조 리서치 | 검색 결과 DB 수집용으로 사용하지 않음 |
| Naver Search API | 필수 | 국내 웹문서, 블로그, 뉴스 후보 수집 | 한국어 제품 스토리 수집에 우선 |
| Firecrawl API | 필수 | 후보 URL 본문을 Markdown/JSON으로 추출 | 자사몰은 자체 크롤러 우선, 외부 URL은 Firecrawl |
| Cloudinary 또는 remove.bg | 선택 | 정밀 배경 제거, 투명 PNG, 고급 자동 크롭 | Supabase Storage Image Transformations만으로 배경 제거는 불가 |

## 3. 제품 목표 및 사용자

### 3.1 목표

1. 200종 전통문화 기념품의 제품 정보, 문화적 배경, 마케팅 문구, 이미지 자산을 한 시스템에서 관리한다.
2. 자사몰 데이터와 외부 웹 검색 자료를 수집하고, 관리자 검수 후 지식베이스에 반영한다.
3. 검수된 지식베이스를 기반으로 목적별, 언어별 콘텐츠를 반자동 생성한다.
4. 원본 제품컷을 기반으로 전통문화 콘셉트의 고품질 연출 이미지를 생성하고 관리한다.
5. 생성물의 출처, 프롬프트, 모델, 승인자, 수정 이력을 추적한다.

### 3.2 사용자 역할

| 역할 | 권한 |
|---|---|
| `admin` | 전체 설정, 사용자 관리, API 상태 확인, 모든 데이터 수정/삭제 |
| `manager` | 제품 등록, 수집 작업 실행, 검수 승인, 콘텐츠/이미지 최종 승인 |
| `editor` | 콘텐츠 생성, 편집, 재생성 요청, 이미지 프롬프트 작성 |
| `reviewer` | 수집 원문 검수, 생성 결과 승인/반려, 피드백 입력 |
| `viewer` | 승인된 제품, 콘텐츠, 이미지 조회 및 다운로드 |

권한은 `auth.users.app_metadata.role` 또는 `public.profiles.role`에 저장한다. `user_metadata`는 사용자가 수정 가능하므로 권한 판단에 사용하지 않는다.

## 4. 전체 사용자 흐름

### 4.1 초기 데이터 구축 흐름

1. 관리자가 `제품 관리 > 제품 가져오기`에서 CSV 또는 수동 입력으로 200종 제품을 등록한다.
2. 시스템이 제품명, 자사몰 URL, 카테고리, 소재, 원본 이미지 유무를 검증한다.
3. 관리자가 `수집 작업 생성`을 실행한다.
4. Edge Function이 Naver Search API로 후보 URL을 수집한다.
5. Firecrawl 또는 자체 크롤러가 본문을 추출한다.
6. 원문은 `검수 대기` 상태로 저장된다.
7. 검수자가 원문을 확인하고 `승인`, `수정 후 승인`, `반려` 중 하나를 선택한다.
8. 승인된 원문만 청킹 및 임베딩 큐에 들어간다.
9. Gemini Embeddings 결과를 Supabase pgvector에 저장한다.
10. 제품 상태가 `지식베이스 준비 완료`로 변경된다.

### 4.2 텍스트 생성 흐름

1. 에디터가 제품 상세에서 `콘텐츠 생성`을 선택한다.
2. 목적, 채널, 언어, 톤, 길이, 금지어, 참고 원문 범위를 선택한다.
3. 시스템이 Supabase pgvector에서 관련 청크를 검색한다.
4. Gemini가 RAG 컨텍스트와 프롬프트 템플릿을 기반으로 초안을 생성한다.
5. 생성 결과는 에디터에서 편집 가능하며, 원문 출처와 함께 표시된다.
6. 리뷰어가 승인 또는 반려한다.
7. 승인된 콘텐츠는 다운로드, 복사, 자사몰 등록용 포맷 내보내기가 가능하다.

### 4.3 이미지 생성 흐름

1. 에디터가 제품 상세에서 원본 제품컷을 업로드한다.
2. 시스템이 파일 형식, 해상도, 배경 제거 가능 여부를 검사한다.
3. 에디터가 콘셉트 템플릿을 선택한다.
4. Gemini 이미지 생성 API가 배경 합성 이미지를 생성한다.
5. 생성 이미지는 갤러리에 `검수 대기` 상태로 저장된다.
6. 리뷰어가 품질 기준을 확인하고 승인 또는 재생성을 요청한다.
7. 승인된 이미지는 SNS 규격별 크롭/리사이즈 후 다운로드 가능하다.

| 우측 상세 | 원문 미리보기, 추출 필드, 검수 폼 |
| 상단 필터 | 제품, 출처 유형, 상태, 도메인, 수집일 |
| 검수 액션 | `승인`, `수정 후 승인`, `반려`, `중복 처리` |

검수 상세 필드:

- 출처 URL
- 출처 유형: `own_mall`, `naver_web`, `naver_blog`, `naver_news`, `manual`
- 제목
- 본문 Markdown
- 추출된 키워드
- 문화/시대/소재 후보
- 신뢰도 점수
- 검수 메모

승인 조건:

1. 출처 URL이 존재하거나 수동 입력 출처 설명이 있다.
2. 본문 길이가 200자 이상이다.
3. 제품과 무관한 내용이 아니어야 한다.
4. 저작권상 그대로 마케팅 문구로 복제할 수 없는 외부 문장은 `참고 자료`로만 저장한다.

### 6.9 콘텐츠 생성 화면

| 영역 | 명세 |

생성 옵션:

| 옵션 | 값 |
|---|---|
| 언어 | 한국어, 영어, 일본어, 중국어 간체 |
| 톤 | 정중한, 감성적, 고급스러운, 간결한, 스토리텔링 |
| 길이 | 300자, 600자, 1000자, 직접 입력 |

사실성 모드 규칙:

- `엄격`: RAG 근거 없는 사실 생성 금지. 출처 없는 역사적 주장 금지.
- `보통`: RAG 근거를 우선하되 일반적 문화 설명 허용.
- `창의적`: 감성 표현 허용. 제품 소재, 원산지, 역사적 사실은 왜곡 금지.

3. 한복, 궁궐, 단청, 한옥 등 문화 요소를 부정확하게 과장하지 않는다.
4. 제품이 배경과 자연스럽게 합성된다.
5. SNS 비율별 크롭 시 제품 핵심부가 잘리지 않는다.

### 10.3 RLS 및 GRANT 기준

모든 `public` 테이블은 RLS를 활성화한다.

```sql
alter table public.products enable row level security;
alter table public.source_documents enable row level security;
alter table public.content_generations enable row level security;
alter table public.image_generations enable row level security;
```

신규 Supabase 프로젝트는 테이블이 Data API에 자동 노출되지 않을 수 있으므로 마이그레이션에 명시적 GRANT를 포함한다.

```sql
grant select on public.products to authenticated;
grant select, insert, update on public.source_documents to authenticated;
grant select, insert, update on public.content_generations to authenticated;
grant select, insert, update on public.image_generations to authenticated;
```

권한 정책 원칙:

- `viewer`: 승인된 데이터만 조회
- `reviewer`: 검수 상태 변경 가능
- `editor`: 콘텐츠/이미지 생성 및 편집 가능
- `manager`: 승인, 반려, 수집 실행 가능
- `admin`: 사용자와 시스템 설정 변경 가능

`service_role` 키는 Edge Functions와 서버 작업에서만 사용한다. 프론트엔드 번들, 브라우저, 클라이언트 로그에 노출하면 안 된다.

## 12. Edge Functions 기준

| 함수 | 역할 |
|---|---|
| `search-sources` | Naver Search API 호출, 후보 URL 저장 |
| `crawl-source` | Firecrawl 또는 자체 크롤러 호출, 본문 추출 |
| `extract-source-metadata` | Gemini로 원문 구조화, 키워드 추출 |
| `embed-product-story` | 승인 원문 청킹, Gemini Embeddings 호출, pgvector 저장 |
| `generate-content` | RAG 검색, Gemini 텍스트 생성, 결과 저장 |
| `generate-image` | 원본 이미지와 프롬프트 기반 이미지 생성 |
| `resize-export-image` | 승인 이미지 SNS 규격별 변환 |
| `retry-job` | 실패 작업 수동 재시도 |
| `audit-event` | 주요 사용자 액션 로그 기록 |

공통 규칙:

- 모든 함수는 idempotency key를 받는다.
- 외부 API 호출 timeout은 기본 60초, 이미지 생성은 최대 180초.
- 응답은 `{ ok: boolean, data?: unknown, error?: { code, message, detail_id } }` 형식.
- 상세 오류는 `audit_logs` 또는 작업 로그에 저장하고 사용자에게는 요약만 표시한다.

## 13. 반응형 기준

본 시스템은 PC 환경 최적화다. 모바일은 조회와 긴급 승인 정도만 허용한다.

| breakpoint | 너비 | 동작 |
|---|---:|---|
| desktop-lg | 1440px 이상 | 기본 3패널 레이아웃 사용 |
| desktop | 1280px 이상 | 기본 지원 |

| mobile | 768px 미만 | 편집/생성 비활성, 조회 중심 |

모바일 제한:

- 이미지 생성 워크벤치 비활성
- 프롬프트 편집 비활성
- 대량 수집 작업 생성 비활성
- 승인/반려, 조회, 다운로드만 허용

## 14. 접근성 원칙

1. 텍스트와 배경 대비는 WCAG AA 기준인 4.5:1 이상을 맞춘다.
2. 모든 버튼, 메뉴, 모달, 탭은 키보드로 조작 가능해야 한다.
3. 포커스 링은 항상 보이게 한다.
4. 아이콘 단독 버튼은 `aria-label`과 tooltip을 제공한다.
5. 상태 색상은 텍스트 라벨과 함께 사용한다.
6. 오류 메시지는 해당 입력 필드와 연결한다.
7. 모달은 focus trap을 적용한다.
8. 테이블 정렬 버튼은 현재 정렬 방향을 스크린리더에 제공한다.
9. 이미지에는 제품명과 용도를 포함한 alt 텍스트를 저장한다.
10. 생성 중 상태는 spinner만 표시하지 않고 `생성 중`, `3/10 처리 중` 같은 텍스트를 병행한다.

## 16. 검수 및 승인 기준

### 16.1 원문 승인 기준

- 제품과 직접 관련된 정보가 포함되어야 한다.
- 출처 URL 또는 수동 출처 설명이 있어야 한다.
- 그대로 복제하면 안 되는 외부 문장은 참고용으로만 저장한다.
- 역사적 사실, 문화재 명칭, 시대명은 검증 가능한 경우에만 승인한다.

### 16.2 콘텐츠 승인 기준

- 제품명, 소재, 용도, 문화적 설명이 제품 DB와 충돌하지 않는다.
- RAG 근거 없는 역사적 사실을 단정하지 않는다.
- 광고 심의상 문제가 될 수 있는 효능, 원산지, 인증 표현을 임의 생성하지 않는다.
- 다국어 콘텐츠는 의미가 원문과 일치해야 한다.

### 16.3 이미지 승인 기준

- 제품 형태가 원본과 다르게 바뀌지 않는다.
- 문화 요소가 부정확하거나 과장되지 않는다.
- 배경에 임의 문자, 왜곡된 문양, 워터마크가 없어야 한다.
- SNS 크롭 규격에서 제품이 잘리지 않아야 한다.

## 17. 보안 기준

1. 모든 외부 API 키는 Supabase Secrets 또는 Vault에 저장한다.
2. 프론트엔드에는 Supabase publishable key만 포함한다.
3. `service_role` 키는 브라우저에 노출하지 않는다.
4. 모든 관리자 작업은 `audit_logs`에 기록한다.
5. 승인 전 원문과 생성물은 외부 공개 URL로 제공하지 않는다.
6. Storage private bucket 파일은 signed URL로만 접근한다.
7. RLS 정책 없이 Data API에 테이블을 노출하지 않는다.
8. 프롬프트에는 비공개 계약 정보, API 키, 내부 계정 정보를 포함하지 않는다.
9. 외부 웹 원문은 생성 근거와 참고 자료로만 사용하고, 원문 복제 콘텐츠를 자동 게시하지 않는다.
